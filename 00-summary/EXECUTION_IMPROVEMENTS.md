## 二、改善方案

### 2.1 Phase 3 實作階段 - 新增檢查點

#### 問題：實作時缺乏「邏輯正確性」自我檢查

**改善**：在實作完成後、提交 Quality Gate 前，增加「邏輯自我檢查」步驟

```markdown
## Phase 3 實作檢查清單（強制）

### 程式碼邏輯檢查
- [ ] 我的輸出會比輸入多嗎？（字串操作）
- [ ] 單一情况與多情况處理一致嗎？
- [ ] 外部依賴是否 lazy check？

### 邊界測試（實作時同步寫）
- [ ] 空輸入
- [ ] 單一元素
- [ ] 邊界值（0, 1, max）
- [ ] **輸出不超過輸入**

### 領域知識對照
- [ ] TTS → 標點是否保留？
- [ ] 網路 → timeout 是否設定？
- [ ] 檔案 → 單一與多是否一致？
```

---

### 2.2 Phase 4 測試階段 - 強化邊界測試

#### 問題：測試只覆蓋正常路徑

**改善**：Test Plan 強制包含「負面測試」

```markdown
## Phase 4 Test Plan 必填項目

### 文字處理邊界測試
- [ ] 空白 → 空陣列
- [ ] 超長 → 正確分割
- [ ] **合併後 ≤ 輸入** ← 關鍵！
- [ ] 無標點 → 不崩潰

### 音訊處理邊界測試
- [ ] 檔案不存在 → FileNotFoundError
- [ ] 單一檔案格式 = 多檔案格式 ← 關鍵！
- [ ] ffmpeg 不可用 → 優雅處理

### 網路處理邊界測試
- [ ] timeout → TimeoutError
- [ ] 連續失敗 → 熔斷觸發
- [ ] 無效設定 → 建構時拋出
```

---

### 2.3 Quality Gate - 新增邏輯正確性檢查

#### 問題：Quality Gate 只檢查「形式」，不檢查「內容」

**改善**：在 Constitution 之外，增加「邏輯正確性」快速檢查

```python
# 自動化邏輯正確性檢查腳本
LOGIC_CHECKS = {
    "字串操作": {
        "pattern": r'\+ ["\'][\。？！\s]',
        "warning": "可能插入額外字符"
    },
    "單一分支": {
        "pattern": r'if len\(.*\) == 1:',
        "warning": "單一情况是否與多情况一致？"
    },
    "初始化依賴": {
        "pattern": r'def __init__.*ffmpeg\.',
        "warning": "外部依賴是否 lazy check？"
    }
}
```

---

### 2.4 實作過程 - 增加「領域知識確認」環節

#### 問題：Agent 缺乏領域知識，導致「看似合理」的錯誤

**改善**：在開始實作前，必須查閱「領域知識清單」

```markdown
## 領域知識確認清單（TTS 專案）

### 實作前必須確認
1. 標點 = 停頓信號 → 合併時不能刪除也不能插入
2. 單一檔案 ≠ 直接複製 → 格式應與多檔案一致
3. ffmpeg 是外部依賴 → 初始化時不能直接崩潰

### 實作中必須檢查
1. 字串操作 → 輸出是否多於輸入？
2. 分支邏輯 → 單一與多是否一致？
3. 外部依賴 → 是否 lazy check？
```

---

### 2.5 開發過程 - 增加「同行邏輯審查」

#### 問題：只有 Constitution 格式檢查，沒有邏輯審查

**改善**：在每個 Phase 完成後，增加「邏輯審查」對話

```markdown
## 邏輯審查對話模板

### Architect 審查問題
1. 「這裡的邏輯有沒有隱式假設？」
2. 「輸出會不會比輸入多？」
3. 「單一情况與多情况一致嗎？」
4. 「這個領域的特殊規則是什麼？」

### Developer 回答
1. 我的假設是：______
2. 輸出預期：______
3. 一致性：______
4. 領域規則：______
```

## 三、執行改善後的流程

### 改善前

```
Phase 3: 實作 → 單元測試 → Quality Gate(形式) → Pass/Fail
                                    ↓
                              邏輯問題漏測
```

### 改善後

```
Phase 3: 實作 → 領域知識確認 → 邏輯自我檢查 → 邊界測試 
            ↓                                          ↓
         單元測試 ←──────────────────────────────── Quality Gate
            ↓                                              ↓
      負面測試 ←────────────── 邏輯正確性檢查 ←─────── Pass/Fail
            ↓
      發現並修復 bug
```

## 四、具體改進清單

| 環節 | 改善項目 | 實作方式 |
|------|----------|---------|
| **實作前** | 領域知識確認 | 強制查閱領域清單 |
| **實作中** | 邏輯自我檢查 | 檢查清單（輸出≤輸入、分支一致、Lazy check） |
| **實作後** | 邊界測試 | 負面測試（空、單一、邊界） |
| **測試 Phase** | 強化 Test Plan | 強制包含輸出≤輸入驗證 |
| **Quality Gate** | + 邏輯檢查 | 自動化邏輯正確性掃描 |
| **審查** | 同行邏輯對話 | Architect 提問 → Developer 回答 |

## 六、總結

**核心改進**：在現有流程中增加「邏輯正確性」檢查環節

| 位置 | 新增檢查 |
|------|----------|
| 實作前 | 領域知識確認 |
| 實作中 | 邏輯自我檢查（輸出≤輸入、分支一致、Lazy check）|
| 測試 Phase | 負面測試（強制邊界測試）|
| Quality Gate | + 自動化邏輯正確性掃描 |
| 審查 | 同行邏輯對話 |

**目標**：讓 bug 在「實作階段」就被發現，而不是等到「測試階段」或「生產環境」
