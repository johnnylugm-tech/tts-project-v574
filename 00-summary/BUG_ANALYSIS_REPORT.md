## 🔴 缺失環節 1：邏輯層面 Spec 映射

### 問題描述
```python
# text_processor.py 第 52 行
sentences = re.split(r'[。？！\n]+', text)
```

**SRS.md FR-05 要求**：「根據句號、問號、驚嘆號進行智能分段」

**實際行為**：將標點作為分割符，但**完全移除**這些標點！

### 開發流程缺失

| 環節 | 應該做什麼 | 實際做了什麼 |
|------|-----------|-------------|
| SRS 閱讀 | 理解「分段」是為了合成，不是為了刪除標點 | 只看到「按標點分割」的表面意思 |
| 邏輯推演 | 「分段後還原」或「保留標點在句尾」 | 假設標點不重要，直接丟掉 |
| Spec 對照 | 畫表格：FR-05 → split() → 應保留標點 | 沒有這個對照表 |

### 根本原因
**Agent 缺乏「標點對 TTS 合成的重要性」領域知識**  
→ 需要在 SKILL.md 或 training 中強調：「中文 TTS 的停頓是靠標點決定的」

## 🔴 缺失環節 2：單元測試覆蓋不足

### 問題描述
測試文件 `test_text_processor.py` 存在，但沒有測試這些邊界情況：

```python
# 應該測試但沒測試的案例
1. ["句子A", "句子B"] → 合併後是否保留原始標點？
2. ["句子A", "句子B"] → 合併時是否多插入了標點？
3. 單一檔案處理是否與多檔案一致？
```

### 開發流程缺失

| 環節 | 應該做什麼 | 實際做了什麼 |
|------|-----------|-------------|
| Test Plan | 列出所有邊界條件 | 只有基本功能測試 |
| Edge Case 設計 | 故意測試「不正常」輸入 | 只測試「正常」情況 |
| 負面測試 | 驗證錯誤處理 | 跳過 |

### 根本原因
**測試設計只追求「覆蓋率數字」，不追求「邏輯正確性」**  
→ Agent Quality Guard 只檢查程式碼結構，不檢查業務邏輯

## 🔴 缺失環節 3：Spec Compliance 驗證

### 問題描述
存在 `verify_spec_compliance.py` 腳本，但：

1. **從未被執行**（或執行了但沒發現問題）
2. **檢查維度不足**：只檢查「有沒有這個功能」，不檢查「功能對不對」

### 開發流程缺失

| 環節 | 應該做什麼 | 實際做了什麼 |
|------|-----------|-------------|
| 自動化驗證 | 每個 commit 都跑 compliance check | 手動，有時忘記 |
| 邏輯驗證 | 輸出「標點保留行為是否符合 FR-05」 | 只有功能存在性檢查 |
| 迴歸測試 | 修改後重新跑所有 spec 檢查 | 沒有這個流程 |

### 根本原因
**把「有沒有」當成「對不對」**  
→ 脚本能檢查「有用 re.split」，但無法檢查「邏輯對不對」

## 🔴 缺失環節 4：集成測試（End-to-End）

### 問題描述
每個模組單獨測試都通過：
- ✅ text_processor.split() → 輸出陣列
- ✅ audio_merger.merge() → 輸出檔案

但**沒測試完整流程**：
```
文字輸入 → 斷句 → 合併 → 音訊輸出
     ↓
  標點在哪裡？
```

### 開發流程缺失

| 環節 | 應該做什麼 | 實際做了什麼 |
|------|-----------|-------------|
| 集成測試 | 文字進 → 音訊出，驗證完整性 | 沒有 |
| 端到端驗證 | 實際播放輸出，確認語速/停頓 | 沒有 |
| 人工驗證 | 開發者實際聽一下輸出 | 沒有（這是 CLI 專案） |

### 根本原因
**缺乏「聽覺驗證」手段**  
→ TTS 專案需要實際播放才能發現問題，但自動化環境無法做到

| 環節 | 應該做什麼 | 實際做了什麼 |
|------|-----------|-------------|

## 📊 開發流程改進建議

### 1. 新增「邏輯層面 Spec 映射」環節

```markdown
## Spec → Code 對照表（強制填寫）

| SRS ID | 需求描述 | 實作函數 | 邏輯驗證 |
|--------|---------|---------|---------|
| FR-05 | 按標點分段，保留標點 | split() | 輸出需含原始標點 |
| FR-07 | 分段不超過 800 字 | _merge_chunks() | 驗證不插入多餘標點 |
```

### 2. 強化「負面測試」要求

```markdown
## Test Plan 必填項目

- [ ] 空白輸入
- [ ] 單一字元
- [ ] 超過 chunk_size 的單一句子
- [ ] 沒有標點的長文本
- [ ] **合併後檢查：是否多於原文？**
```

### 3. 增設「 Spec Logic Check 」自動化

```python

    """驗證：輸出不應比輸入多任何字符"""
    for chunk in result:
        # 檢查：chunk 中的字符是否都來自原始 text
        pass
```

### 4. 建立「領域知識清單」

```markdown
## TTS 領域知識（SKILL.md 需包含）

1. 標點 = 停頓信號，刪除會讓合成不自然
2. 語速 -2% = 略慢，适合简报
```

## 🎯 結論

| 問題 | 根因 | 預防方案 |
|------|------|---------|
| `_merge_chunks` 多餘句號 | split() 丟掉標點，未還原 | Spec 邏輯映射 + 邊界測試 |
| 單檔案直接 copy | 沒有「格式一致性」概念 | 集成測試 + 領域知識 |
| ffmpeg 初始化崩潰 | 沒有「Lazy check」思維 | Code Review 盲點彌補 |

**核心教訓**：Quality Gate 只能檢查「形式合规」，無法保證「逻辑正确」。需要领域知识 + 集成测试 + 人工review的组合来补强。
